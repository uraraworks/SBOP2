# SboCliAdminMfc — アーカイブ宣言

## 状態

このプロジェクトは **2026-04-09 時点でビルド不能**であり、ソリューションのビルド構成からも除外されています（プロジェクト自体は SBO.sln に残っていますが Build.0 エントリを削除済み）。**今後復旧する予定はありません**。

## 経緯

- **Phase 4** (`ae98953`, MFC 完全除去) で SboCli 本体から MFC への依存を取り除いた
- その時点で `SboCliAdminMfc/StdAfx.h` が `SboCli/StdAfx.h` を共有 include しているだけだった構造により、AdminMfc 側 PCH から `<afxwin.h>` が消失
- 以後、`AdminWindow.h:54` の `class CAdminWindow : public CWnd, ...` を起点に `CWnd` 未定義の連鎖エラー（C2065 が約 270 件）が発生
- **2026-04-09** にユーザーがビルドして 100 件超のエラーで停止することを確認
- 同日、復旧コストと将来戦略を検討した結果「凍結 + Web 化」方針を採用

## なぜ復旧しないか

管理者機能は **最終的に Web フロントエンドに置き換える** 方針です。

- AdminMfc を MFC のまま復旧 → 数週間後に廃止予定なので投資の無駄
- SDL 化（MFC 除去）→ 規模が大きく、しかも最終形ではない
- Web 化 → OS 非依存・リモート管理・認証統合などの本質的メリットがあり、これが最終ゴール

短期的に AdminMfc を「動く管理 UI」として復旧する技術的価値より、Web 化に資源を集中する方が合理的と判断しました。

## このプロジェクトの参考価値

ソースコード自体は **Web 化のための仕様書代わり**として保持します。具体的には:

| ディレクトリ / ファイル | 参考になるもの |
|---|---|
| `src/Migrated/Dialog/` 配下 50+ ダイアログ | 管理者操作のユースケース一覧（キャラクタ管理・アカウント管理・アイテム編集・マップ編集 など） |
| `src/Migrated/Wnd/` 配下 | リスト/ツリー UI の項目構造 |
| `AdminApi/` (SboCli 側) と DLL の境界 | 管理パケットの呼び出しパターン、`AdminUiLoader.cpp` 経由の動的ロード設計 |
| `AdminWindow.h` / レイアウトヘルパー | 各ダイアログの入力フィールド構成 |

## 既知の腐敗リスク

SboCli を改修するたびに、AdminMfc が共有 include している `Common/` `SboCli/src/` 配下のファイルとの不整合が増えていきます。これは**容認します**。参考資料として「ファイルが何を表しているか」が読めれば十分で、コンパイルを通す必要はありません。

## ファイルツリー構造を保持する理由

ビルド不能でも以下の理由でディレクトリは残します:

1. git history による変更追跡が可能
2. 機能仕様の参照ソース（Web 化時の対比対象）
3. 削除した瞬間に「何を置き換えるべきか」が分からなくなる

## 復旧不能になっている主要エラー（参考）

| エラーコード | 件数 | 主因 |
|---|---|---|
| C2065 | 約 272 | `CWnd` 等の MFC クラス未定義（連鎖エラー） |
| C4430 | 約 204 | 同上由来の型指定子なし |
| C2872 | 約 120 | 名前空間あいまい |
| C2660 | 約 81 | スクロール関数等の MFC API 引数不一致 |
| C1083 | 約 73 | `Migrated/Wnd/` 配下の欠落ヘッダ（過去削除済み） |
| C3861 | 約 34 | `SdlDCRegister`/`SdlDCUnregister` 未定義 |
| C2504 | 約 11 | 未定義の基底クラス（`CWnd`） |

起点はいずれも 1〜数か所ですが、復旧は **段階1〜4 で 5〜10 時間規模** と見積もられます（2026-04-09 時点）。

## Web 化に向けた論点（詰めるべき項目）

Web 化を本格着手する前に、以下を一度整理しておく必要があります:

| 論点 | 質問 |
|---|---|
| バックエンド配置 | サーバー本体 (`SboSvr`) に管理 API を生やす？ それとも別プロセス？ |
| プロトコル | 既存 UraraSockTCP の管理パケットを Web から呼ぶ（プロキシ）？ それとも REST/gRPC を新設？ |
| 実装言語 | バックエンドは C++ 継続？ それとも別言語で薄く包む？ |
| フロントエンド | SPA (React/Vue)？ サーバーサイドレンダリング？ |
| 認証 | 既存の管理者アカウント体系をそのまま使う？ 新規？ |
| 移行範囲 | 50+ ダイアログ全部 Web 化？ それとも段階的に主要機能から？ |

## 削除予定

Web 化された管理 UI が本番運用に乗った時点で、このプロジェクトとそれに紐づくファイル群は完全削除します。それまでは「凍結された参考資料」として扱ってください。
