# ブラウザでタイトル画面表示まで到達するための実装手順

## 0. 2026-04-01 時点の進捗

### 到達できたこと

- Emscripten で browser build が通る
- 専用 shell で `sbocli-title.html/js/wasm/data` を配信できる
- `SboGrpData/res` を preload して、browser 側から title 用素材を参照できる
- `GAMESTATE_LOGO` まで進み、ロゴ画面が browser/headless Chrome で表示される
- ロゴの fade-in / fade-out も browser 側で進むことを headless 画像で確認できる
- `GAMESTATE_LOGIN` への状態遷移自体は発生している

### まだ未完了のこと

- `GAMESTATE_LOGIN` 遷移後のタイトル画面表示は未安定
- headless ではロゴ後に黒画面のまま止まるケースが残っている
- browser では Win32 メッセージ順序に依存した初期化・通知順がまだ残っている

### 現時点の評価

このマイルストーンは「ロゴ表示まで」は実現済みで、残りの主課題は描画 API そのものよりも、

- `WM_MAINFRAME` / `WM_MGRDRAW` 依存
- Win32 ウィンドウ存在前提の通知順
- browser 向けフレーム供給と再描画要求

を整理することに移っている。

### 次セッションの方針

次は予定どおり、タイトル画面のその場しのぎ修正を増やすのではなく、**Windows 依存の縮退を先に進める**。
特に `MainFrame` / `MgrDraw` / `StateProc*` 間の通知を、Win32 メッセージから内部イベントまたは直接呼び出しへ寄せる。

## 1. 目的

この計画の目標は、**サーバー通信なしで、ブラウザ上で `SboCli` 相当のクライアントを起動し、ロゴ画面からタイトル画面まで表示できること** である。

この段階では、以下は対象外とする。

- サーバー接続
- ログイン送信
- キャラ選択
- マップ入場
- WebSocket
- `clientId`

つまり、最初のブラウザ到達目標は以下である。

1. WebAssembly ビルドできる
2. ブラウザで起動できる
3. ロゴが出る
4. タイトル背景が出る
5. BGM が鳴る、または少なくとも音声初期化で落ちない

---

## 2. 完了時の到達イメージ

以下が通れば、このマイルストーンは完了とする。

1. ブラウザでアプリを読み込める
2. 初期化失敗で即終了しない
3. `GAMESTATE_LOGO` に入る
4. ロゴのフェードイン/フェードアウトが動く
5. `GAMESTATE_LOGIN` に遷移する
6. タイトル背景が表示される
7. ログイン入力欄が未実装またはダミーでも、タイトル表示までクラッシュしない

---

## 3. 前提整理

ブラウザでタイトル表示まで進めるためには、次の 3 層を順に片付ける必要がある。

### A. ネイティブ起動でタイトル画面まで通す

まず Windows 上の SDL 実行で、

- 起動
- ロゴ
- タイトル

の流れを安定させる。

### B. ブラウザで使えない Windows 依存をタイトル到達経路から外す

特に以下が対象になる。

- `HWND`
- Win32 メッセージループ
- `CreateWindow("EDIT"/"BUTTON")`
- `Imm*`
- GDI / `HDC`
- DirectInput
- Windows 専用音声 API

### C. Emscripten / WebAssembly ビルドに通す

そのうえで、

- SDL + Emscripten ビルド
- ブラウザ上でのイベントループ
- 画像・音声アセット読み込み

を成立させる。

---

## 4. 基本方針

### 4.1 まず「タイトル到達に必要なもの」だけ通す

最初から全機能をブラウザ対応しない。

優先するのは次のみ。

- `MainFrame`
- `StateProcLOGO`
- `StateProcLOGIN`
- `MgrDraw`
- `MgrLayer`
- `LayerLogo`
- `LayerTitle`
- タイトル到達に必要な最小限の `MgrSound`

### 4.2 `WindowLOGIN` は最初は完成させない

`WindowLOGIN` は Win32 の `EDIT` / `BUTTON` に強く依存しているため、この段階では次のどちらかにする。

- ブラウザビルド時は生成しない
- ダミー実装へ差し替える

ここで重要なのは、**タイトル画面表示の阻害要因から外すこと** である。

### 4.3 通信は完全に無効化する

ブラウザタイトル表示段階では、接続処理に入らないようにする。

必要ならビルドフラグや実行モードで、

- ローカルタイトル表示専用モード

を作る。

### 4.4 `#ifdef` より抽象化を優先する

この段階の実装方針として、**ゲームロジックや状態遷移層へ `#ifdef` を散らさない** ことを原則とする。

基本方針は次のとおり。

1. 差し替えが必要な機能はインターフェイス化する
2. デスクトップ版とブラウザ版の差分は、それぞれの実装クラスへ閉じ込める
3. `#ifdef` を使う場合でも、できるだけ実装末端に限定する

### 4.5 この段階で優先して抽象化する対象

タイトル画面表示までの段階で、特に抽象化候補になるのは以下。

| 対象 | 推奨抽象 |
|---|---|
| ログイン UI | `ILoginView` |
| 音声初期化/再生 | `IAudioService` |
| 起動モード | `IRuntimeMode` または設定オブジェクト |
| 将来の通信 | `ITransport` |

この段階では、全部を一気に抽象化する必要はない。  
まずは **`WindowLOGIN` を `ILoginView` 相当へ切り出すこと** が最優先である。

### 4.6 `WindowLOGIN` の推奨分割

現状の `WindowLOGIN` は、

- UI の見た目
- 入力欄
- ボタン
- フォーカス移動
- 接続通知

が 1 クラスに混在している。

これを今後は次のように分けるのが望ましい。

- 共通インターフェイス: `ILoginView`
- デスクトップ実装: `Win32LoginView`
- ブラウザ向け暫定実装: `NullLoginView` または `BrowserTitleStubLoginView`
- 将来の本実装: `SdlLoginView` または `BrowserLoginView`

タイトル表示マイルストーンでは、まず `NullLoginView` 相当で十分である。

### 4.7 `#ifdef` を使ってよい場所

全面禁止ではなく、以下は許容する。

- 実装クラス内部
- ビルド設定ファイル
- プラットフォーム固有のファクトリ生成箇所

逆に避けるべきなのは次。

- `StateProcLOGO`
- `StateProcLOGIN`
- `MainFrame` の状態遷移ロジック
- 共通 `Mgr*` クラスの高レベル制御

つまり、

- `if browser then ... else ...`

のような分岐をゲーム進行コードへ直接書かず、

- `CreateLoginView()` が適切な実装を返す

形に寄せる。

## 5. 実装手順

以下を、そのまま作業順として使う。

### Step 1: タイトル到達の最小経路を固定する

**目的**

- タイトル表示までに本当に必要なファイルと処理を確定する

**作業**

- `MainFrame::OnInitEnd()` から `GAMESTATE_LOGO` までの経路確認
- `StateProcLOGO` と `StateProcLOGIN` の依存を確認
- `MgrLayer::MakeLOGO()` / `MakeTITLE()` が必要とする素材とマネージャ確認
- 「この段階では不要なもの」を明示的に列挙

**成果物**

- タイトル到達の必須依存一覧
- タイトル到達の不要依存一覧

**完了条件**

- 以降の作業で対象範囲がぶれない

### Step 2: ネイティブ実行で「起動 -> ロゴ -> タイトル」を安定化する

**目的**

- ブラウザ移植前に、SDL 実行で最小経路が安定している状態を作る

**作業**

- ロゴ画面表示確認
- フェード処理確認
- タイトル背景表示確認
- タイトル BGM 初期化確認
- `WindowLOGIN` がタイトル表示を壊す場合は切り離す

**方針**

- ここではブラウザ対応ではなく、まず「ローカル無通信で安定表示」がゴール

**完了条件**

- Windows 上で通信なしでもタイトル画面まで安定到達できる

### Step 3: タイトル到達経路から Win32 入力欄依存を外す

**目的**

- `WindowLOGIN` の Win32 コントロール依存を、ブラウザタイトル表示のブロッカーから外す

**作業**

- `WindowLOGIN` を直接使う構造を見直し、`ILoginView` 相当へ切り出す
- `StateProcLOGIN::Init()` の `MakeWindowLOGIN()` 依存をファクトリ経由へ寄せる
- ブラウザ向けビルドでは `NullLoginView` などのダミー実装を使い、タイトル背景だけ出せるようにする
- `TimerProc()` がダミー実装でも落ちないよう整理する

**実装案**

- `ILoginView`
- `Win32LoginView`
- `NullLoginView`

を導入し、生成はファクトリまたは `MgrWindow` 側で切り替える

最小構成では、ブラウザ向けは `NullLoginView` を返せばよい

**完了条件**

- `WindowLOGIN` がなくても `GAMESTATE_LOGIN` でタイトル背景表示まで進める

### Step 4: タイトル到達経路から通信依存を外す

**目的**

- ブラウザ段階でサーバー通信に触れないようにする

**作業**

- タイトル表示までに `m_pSock->Connect()` や通信初期化が走らないことを確認
- ローカルタイトルモードでは接続ボタンや接続通知を無効化
- 必要なら接続関連メニューや遷移を抑止

**完了条件**

- ブラウザビルドで通信層がなくてもタイトル表示まで到達できる

### Step 5: タイトル到達経路の Win32 メッセージ依存を整理する

**目的**

- ブラウザで使えない Window メッセージ依存のうち、タイトル到達に関係するものを局所化する

**作業**

- `WM_MAINFRAME`
- `WM_MGRDRAW`
- 必要なら `WM_INITEND`

のうち、タイトル到達に必要な通知を確認する

- SDL 実行下で代替できるものは整理
- ブラウザでも維持する必要があるなら、内部通知として抽象化する

**この段階で重点的に見る箇所**

- `StateProcLOGO::OnMgrDrawEND_FADEOUT()`
- `StateProcLOGIN::OnMgrDrawSTART_FADEIN()`
- `MainFrame::OnMainFrame()`
- `MainFrame::OnMgrDraw()`

**完了条件**

- タイトル到達に必要な通知が、Win32 にべったりでなく追跡可能になっている

### Step 6: タイトル到達に必要な描画経路を Emscripten 対応観点で切り出す

**目的**

- GDI ベース描画のうち、ロゴ・タイトル表示に必要な部分だけでもブラウザで動かせる見通しを作る

**作業**

- `LayerLogo`
- `LayerTitle`
- `MgrDraw::Draw()`

で使っている `CImg32` / `HDC` / `TextOut` 依存の確認

次の 2 つのどちらで行くか決める。

- A. タイトル表示までは既存 `CImg32` を SDL テクスチャ更新で暫定運用
- B. ロゴ/タイトルだけ先に SDL ネイティブ描画へ差し替える

**推奨**

- 最初は A を推奨
- つまり、内部バッファは暫定維持しつつ、Emscripten で問題になるところだけを潰す

**完了条件**

- タイトル表示に必要な描画経路が「ブラウザでは絶対無理」という状態でなくなる

### Step 7: タイトル到達に必要な音声経路を整理する

**目的**

- BGM や効果音が原因でブラウザ起動が止まらないようにする

**作業**

- `MgrSound` の初期化経路確認
- タイトル表示に必要な BGM/SE のみ抽出
- ブラウザで使えない音声 API がある場合:
  - タイトル段階では無音フォールバック
  - または SDL 音声経路へ仮置き

**完了条件**

- 音が鳴らなくても、音声初期化失敗でタイトル表示が止まらない

### Step 8: Emscripten ビルド構成を追加する

**目的**

- ブラウザ向けに実際にビルドできるようにする

**作業**

- Emscripten 用ビルド設定追加
- 必要な SDL オプション設定
- タイトル専用ビルドフラグ追加
- 不要機能のコンパイル除外または stub 化

**完了条件**

- WebAssembly / HTML / JS 出力が得られる

### Step 9: ブラウザ起動時のアセット読み込みを成立させる

**目的**

- ロゴ画像、タイトル画像、BGM などの必要最低限アセットをブラウザで読めるようにする

**作業**

- どのアセットがタイトル到達に必要か確認
- パッケージ方法を決める
  - preload
  - 埋め込み
  - 仮想 FS
- パス解決をブラウザで通す

**完了条件**

- ブラウザでタイトルに必要なアセットが見つかる

### Step 10: ブラウザで実機確認する

**目的**

- 実際にブラウザで起動してタイトルまで進むことを確認する

**作業**

- ロゴ表示確認
- フェード確認
- タイトル背景確認
- BGM または無音フォールバック確認
- `WindowLOGIN` 未実装でもクラッシュしないこと確認

**完了条件**

- ブラウザでタイトル画面が表示される

## 7. マイルストーン

### M1: ネイティブでタイトル画面まで到達

- Windows 上で通信なしタイトル表示

### M2: `WindowLOGIN` と通信依存の切り離し

- タイトル表示経路を単独化

### M3: Emscripten ビルド通過

- ブラウザ用出力生成

### M4: ブラウザでロゴ表示

- 起動してロゴが見える

### M5: ブラウザでタイトル表示

- ロゴからタイトルへ遷移する

### M6: ブラウザタイトル表示の安定化

- 複数回再現する

---

## 8. GO判断用の条件

将来「最後まで進めてください」で実装着手するための前提として、以下が定義済みであればよい。

1. 最初の目標が「ブラウザでタイトル画面表示まで」で確定している
2. `WindowLOGIN` は最初は未完成でもよいと合意できている
3. 通信はこの段階では対象外と合意できている
4. タイトル到達に必要なファイル群を優先して進めると決めている
5. `#ifdef` を高レベルロジックへ散らさず、差し替え実装で進める方針に合意できている

この 5 点が合っていれば、この計画に沿って最後まで進めやすい。

---

## 9. 受け入れ基準

- ブラウザでビルド成果物を読み込める
- アプリが起動時に落ちない
- ロゴが表示される
- フェードアウト後にタイトル画面へ遷移する
- タイトル背景が見える
- 音声が鳴る、または音声なしでも止まらない
- サーバー通信なしで成立する
- `WindowLOGIN` の Win32 入力欄が未対応でもタイトル表示までクラッシュしない

## 補足メモ

2026-04-01 時点では `M4: ブラウザでロゴ表示` までは到達済み。  
`M5: ブラウザでタイトル表示` は、遷移そのものは起きているが、描画経路と通知順の Windows 依存が残っているため未完了。
